H5 软键盘兼容方案

在 App 内嵌 WebView 的聊天、评论、搜索等场景中，页面通常由三部分组成：顶部 Header（固定）、中间对话区（可滚动）、底部输入框（固定）。

理想状态下，软键盘弹出时只有中间对话区被压缩，Header 始终吸顶，输入框随键盘上推。但在 iOS 和 Android 上，实际情况往往与预期不符。本文整理了实际开发中常见的兼容问题，以及对应的处理方案。

一、理想布局

核心诉求可以概括为：键盘只影响中间内容区的高度，不改变 Header 和输入框的相对位置关系。

二、iOS

问题一：fixed 定位的 Header 跟着页面上滑

现象

输入框获得焦点后，Header 虽然设置了 position: fixed，但键盘弹出时仍会跟着整个页面一起滑走，无法保持吸顶。

原因

2015 年 3 月 iOS 8.2 发布后，键盘弹起时 WebView 不会被压缩，而是整体被顶上去。此时 fixed 元素不会重新计算定位，仍保持原来的相对位置，于是跟着页面一起上移。

解决方案

方案一：客户端原生 Header（推荐）

由客户端提供自定义 Header，或针对当前场景定制原生导航栏。原生 Header 不属于 WebView 内容，WebView 上推时它依然保持吸顶——这是从根源上规避问题的方式。

方案二：H5 动态修正 Header 定位

既然页面高度没变、只是整体上移，那么获取键盘高度后重新设置 Header 的 top 即可。

已知缺点

切换键盘类型（如中英文切换）时，无法检测到键盘高度变化，页面会有轻微上下跳动。如需更平滑，可在 focus 期间监听页面滚动，实时更新 Header 的 top，blur 后恢复。

方案三：H5 + 客户端 SDK

客户端提供 SDK，实时通知前端键盘高度变化，前端据此动态设置 top。精度和实时性最好，但需要客户端配合。

三、Android

Android 不会出现 Header 不固定的问题，但键盘弹出的表现分两种模式。

正常模式

键盘弹出时 WebView 被压缩，底部输入框自然上滑——最理想的状态，无需额外处理。

异常模式（华为等机型居多）

键盘弹出后页面毫无变化，WebView 没有被压缩，键盘直接覆盖在 H5 页面上，输入框被挡住。

原因尚不明确，且因为可视区没有变化，前端也无法直接获取键盘高度。

解决方案

方案一：H5 按屏幕比例估算键盘高度

思路比较粗糙：根据不同设备的宽高比推算键盘大概高度，给一个固定值。以下系数来自常见机型的实测经验，上线前需按目标设备微调。

  • scrollIntoView 也无效——系统认为屏幕大小就是可视区，不会产生滚动

方案二：H5 + 客户端 SDK

与 iOS 方案三相同，由客户端实时通知键盘高度，前端动态调整布局——最理想的长期方案。

五、总结

H5 软键盘兼容没有银弹，核心思路是：

  1. 能交给客户端的尽量交给客户端——原生 Header 或键盘高度 SDK 是最稳的方案
  2. iOS 的关键是补偿 pageYOffset——键盘顶起整个页面后，用 top 把 fixed 元素拉回来
  3. Android 异常机型只能曲线救国——按屏幕比例估算，先过滤正常机型再处理
  4. 避开 scrollTo + -webkit-overflow-scrolling: touch 的组合——会引发滚动冲突

如果业务对体验要求高，建议推动客户端提供键盘高度 SDK，前端统一监听、统一适配，一劳永逸。
